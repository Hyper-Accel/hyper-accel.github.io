안녕하세요, HyperAccel의 ML 팀에서 일하고 있는 엄태서입니다.

지난 글에서는 우리가 어떻게 **Copy & Paste** 시대에서 코딩 에이전트 시대로 넘어왔는지 이야기했습니다. 처음에는 AI가 조금 똑똑한 검색창처럼 보였습니다. 그다음에는 자동완성이 되었고, 이제는 `plan.md` 를 만들고 테스트를 돌리며, 실패하면 다시 시도하는 동료에 가까워졌습니다.

이번 글의 주인공은 조금 의외입니다. 더 화려한 IDE도 아니고, 더 예쁜 채팅 UI도 아닙니다. 바로 터미널입니다. 네, 그 검은 화면입니다. 한동안 “언젠가 사라질 레거시 인터페이스”처럼 취급받던 **Command Line Interface(CLI)** 가 바이브 코딩 시대의 가장 뜨거운 무대로 돌아오고 있습니다.

재미있는 점은 이유가 향수 때문이 아니라는 것입니다. 개발자들이 갑자기 1990년대 감성에 빠진 것이 아닙니다. AI 에이전트가 가장 자연스럽게 숨 쉬는 곳이 터미널이기 때문입니다.

## 모델 전쟁에서 하네스 전쟁으로

지난 1~2년 동안 우리는 거의 스포츠 중계처럼 모델을 비교했습니다. “GPT가 이겼다”, “Claude가 더 잘 짠다”, “Gemini가 길게 기억한다.” 물론 모델 성능은 중요합니다. 엔진이 약한 스포츠카는 멀리 못 갑니다.

하지만 에이전트 시대에 들어오면 질문이 조금 바뀝니다.

**“어떤 모델이 가장 똑똑한가?”** 보다 더 중요한 질문은 **“그 모델을 어떤 환경에 넣었는가?”** 입니다.

저는 이것을 **Harness Wars** 라고 부르고 싶습니다. 여기서 하네스는 모델을 둘러싼 실행 환경입니다. 파일을 읽고, 명령어를 실행하고, 테스트를 돌리고, 실패 로그를 다시 해석하고, 변경 사항을 안전하게 되돌릴 수 있는 구조입니다.

좋은 하네스 안에 들어간 평범한 모델은, 아무 도구 없이 빈 채팅창에 앉아 있는 프론티어 모델보다 더 실용적일 때가 많습니다. 개발은 시험장에서 정답을 쓰는 일이 아니라, 지저분한 레포지토리 안에서 계속 피드백을 받으며 문제를 줄여 나가는 일이기 때문입니다.

이 차이는 생각보다 큽니다. 채팅창의 모델은 “답변”을 합니다. 하네스 안의 에이전트는 “작업”을 합니다. 그리고 소프트웨어 엔지니어링에서 우리는 대개 예쁜 답변보다 통과하는 작업물을 원합니다.

## 왜 다시 CLI인가

터미널은 원래부터 개발자의 백스테이지였습니다. 빌드, 테스트, 배포, 로그 확인, `git diff`, `grep`, `curl`, `ssh`. 우리가 IDE에서 아무리 많은 시간을 보내도, 실제 시스템의 맥박은 터미널에 남아 있었습니다.

AI 에이전트는 바로 그 맥박을 좋아합니다.

CLI는 빠릅니다. UI 버튼을 찾을 필요가 없습니다. 명령은 텍스트이고, 출력도 텍스트입니다. 에이전트 입장에서는 이것이 거의 모국어에 가깝습니다. `npm test` 를 실행하고, 실패 로그를 읽고, 관련 파일을 열고, 패치를 만들고, 다시 테스트합니다. 이 루프는 마우스 클릭보다 쉘 파이프에 훨씬 잘 맞습니다.

CLI는 조합 가능합니다. `grep` 의 출력은 다음 명령의 입력이 되고, 테스트 로그는 다음 추론의 컨텍스트가 됩니다. 사람에게는 오래된 유닉스 철학처럼 보이지만, 에이전트에게는 훌륭한 감각기관입니다.

CLI는 날것의 제어권을 줍니다. IDE가 친절한 도시라면, 터미널은 공항 관제탑에 가깝습니다. 복잡하지만 직접적입니다. 에이전트가 실제로 무엇을 했는지, 어떤 명령을 실행했는지, 어디서 실패했는지 추적하기 쉽습니다.

그래서 터미널 기반 에이전트 워크플로우는 이상하게도 “미래적”이면서 “원초적”입니다. 검은 화면에 앉아 자연어로 목표를 던지면, 그 아래에서는 파일 시스템, 테스트 러너, 패키지 매니저, Git이 조용히 움직입니다. 개발자가 직접 타이핑하던 명령들이 이제는 에이전트의 팔과 다리가 됩니다.

이것이 제가 생각하는 바이브 코딩의 실제 모습입니다. 대충 코딩한다는 뜻이 아닙니다. 생각과 말의 속도에 맞춰 작업 단위를 던지고, 에이전트가 실행 가능한 루프로 바꿔 주는 방식입니다. 개발자는 키 입력 노동에서 조금 빠져나와, 방향과 검증에 더 많은 시간을 쓰게 됩니다.

## IDE는 사라지지 않는다. 역할이 바뀔 뿐이다

여기서 오해하면 안 됩니다. 이 글은 “IDE는 끝났다”는 선언이 아닙니다. IDE는 여전히 훌륭합니다. 리팩터링, 디버깅, 타입 탐색, 시각적 코드 탐색, 마지막 폴리싱에는 IDE만큼 편한 도구가 많지 않습니다.

다만 에이전트 시대에는 무게중심이 조금 움직입니다.

IDE는 정교한 조각칼에 가깝습니다. CLI 에이전트는 작업장 전체를 움직이는 포크리프트에 가깝습니다. 하나는 손끝의 감각을 살리고, 하나는 큰 덩어리를 옮깁니다. 둘 중 하나만 선택해야 하는 문제가 아니라, 어떤 단계에서 어떤 도구를 쓰느냐의 문제입니다.

저는 이렇게 나눠서 생각합니다.

| 구분 | Model Era | Agentic Era |
| --- | --- | --- |
| 중심 질문 | 어떤 모델이 더 똑똑한가 | 어떤 하네스가 더 일을 잘 굴리는가 |
| 인터페이스 | 채팅창, 자동완성 | CLI, 도구 호출, 실행 루프 |
| 개발자 역할 | 프롬프트 작성자 | 컨텍스트 설계자, 리뷰어, 오케스트레이터 |
| 산출물 | 답변, 코드 조각 | 패치, 테스트 결과, 실행 로그 |
| 핵심 기술 | Prompt Engineering | Context Engineering |

IDE는 코드를 polishing하는 공간으로 남습니다. CLI와 에이전트는 코드를 manifesting하는 공간이 됩니다. 무언가를 눈앞에 불러내는 일, 즉 “이 기능을 만들어 봐”, “이 테스트 실패를 끝까지 고쳐 봐”, “이 플랜대로 PR 단위 작업을 나눠 봐” 같은 명령은 터미널에서 훨씬 자연스럽습니다.

## Prompt Engineering에서 Context Engineering으로

한동안 우리는 좋은 프롬프트를 쓰는 법을 배웠습니다. “역할을 부여하세요”, “단계적으로 생각하게 하세요”, “예시를 주세요.” 이 조언들은 여전히 유효합니다. 하지만 에이전트 시대의 진짜 기술은 프롬프트 문장 하나를 예쁘게 쓰는 것이 아닙니다.

이제 중요한 것은 **Context Engineering** 입니다.

컨텍스트 엔지니어링은 에이전트가 어떤 상태에서 일하게 할지 설계하는 일입니다. 어떤 파일을 읽게 할지, 어떤 규칙을 기억하게 할지, 어떤 테스트를 신뢰하게 할지, 실패했을 때 어디까지 다시 시도하게 할지 정하는 일입니다.

사람으로 비유하면 이렇습니다. 프롬프트 엔지니어링은 주니어 엔지니어에게 “좋은 코드로 짜 주세요”라고 말하는 법을 개선하는 일입니다. 컨텍스트 엔지니어링은 그 주니어에게 온보딩 문서, 아키텍처 다이어그램, 테스트 계정, CI 규칙, 리뷰 기준을 제대로 쥐여 주는 일입니다.

당연히 후자가 더 중요해집니다.

에이전트는 기억력이 좋은 것처럼 보이지만, 실제로는 작업 상태를 계속 잃어버릴 수 있습니다. 어떤 결정이 이미 내려졌는지, 어떤 파일은 건드리면 안 되는지, 어떤 테스트 실패는 기존 flaky 이슈인지 모르면 그럴듯한 삽질을 아주 빠르게 합니다. 속도가 빠르다는 것은 장점이지만, 방향이 틀리면 더 빠르게 절벽으로 갑니다.

그래서 좋은 에이전트 워크플로우에는 명시적인 상태 관리가 필요합니다. 계획 파일, 체크리스트, 실행 로그, 테스트 증거, 변경 범위, 금지 사항. 예전에는 이것이 프로세스 문서처럼 보였지만, 이제는 에이전트의 작업 기억장치가 됩니다.

## 연구도 같은 방향을 가리킨다

이 흐름은 단순한 개발자 취향의 변화만은 아닙니다. 연구와 벤치마크도 비슷한 방향을 보여 줍니다. 모델에게 한 번에 “정답 코드”를 쓰게 하는 방식보다, 모델이 파일을 읽고, 명령을 실행하고, 실패를 관찰하고, 다시 고치는 루프가 실제 개발에 더 가까워지고 있습니다.

예를 들어 [Software Engineering Benchmark(SWE-bench)](https://www.swebench.com/original.html)는 실제 GitHub 이슈를 기반으로 모델이 코드베이스를 패치할 수 있는지 평가합니다. [SWE-bench Verified](https://www.swebench.com/verified)는 그중 사람이 검수한 500개 태스크를 사용합니다. 여기서 중요한 것은 특정 점수 하나가 “코딩 지능”을 증명한다는 뜻이 아닙니다. 오히려 소프트웨어 엔지니어링 평가는 점점 더 “답변 생성”이 아니라 “환경 안에서 고치기”에 가까워지고 있다는 점입니다.

Andrew Ng도 에이전트 워크플로우를 이야기할 때 **Reflection**, **Tool Use**, **Planning**, **Multi-agent Collaboration** 같은 패턴을 강조합니다. 이것은 경험적 만능 법칙이라기보다, 우리가 어떤 방향으로 도구를 설계하고 있는지 보여 주는 좋은 지도입니다. 모델에게 한 번 묻고 끝내는 것이 아니라, 모델이 도구를 쓰고, 결과를 비판하고, 계획을 세우고, 여러 역할을 조합해 문제를 푸는 구조가 중요해지고 있습니다.

[CodeAct](https://arxiv.org/abs/2402.01030) 같은 연구도 이 맥락에서 흥미롭습니다. CodeAct는 에이전트의 행동 공간을 텍스트나 JSON 호출이 아니라 실행 가능한 코드로 표현합니다. 논문은 특정 도구 사용 벤치마크에서 코드 기반 액션 포맷이 텍스트/JSON 기반 포맷보다 더 좋은 결과를 낼 수 있음을 보고합니다. 물론 이것을 “항상 코드 액션이 이긴다”로 해석하면 안 됩니다. 결과는 벤치마크, 도구 접근성, 실행 환경에 크게 의존합니다.

하지만 안전하게 말할 수 있는 것은 있습니다. 소프트웨어 엔지니어링과 도구 사용 영역에서는 **모델 단독 성능** 만큼이나 **루프, 도구, 실행 환경, 컨텍스트 관리** 가 중요해지고 있습니다.

## 유명해진 하네스들은 무엇을 잘했나

그래서 요즘 개발자 도구를 보면 모델 이름보다 하네스의 성격이 더 잘 보입니다. 같은 Claude나 GPT 계열 모델을 쓰더라도, 어떤 하네스는 페어 프로그래머처럼 행동하고, 어떤 하네스는 GitHub 이슈 해결 로봇처럼 움직이고, 어떤 하네스는 여러 전문 에이전트를 동시에 굴리는 작은 조직처럼 동작합니다.

[Claude Code](https://code.claude.com/docs/en/overview)는 이 흐름에서 가장 상징적인 기준점 중 하나입니다. 터미널 안에서 레포지토리를 읽고, 파일을 수정하고, 명령을 실행하며, **Model Context Protocol(MCP)** 과 hooks, subagents 같은 확장 지점을 제공합니다. 유명해진 이유는 단순히 “Claude를 CLI에 넣었다”가 아닙니다. 개발자가 이미 일하던 검은 화면 안에서 모델이 실제 작업 루프에 들어왔기 때문입니다.

[OpenCode](https://opencode.ai/)는 비슷한 터미널 감각을 더 열려 있는 방향으로 밀어붙입니다. provider-agnostic하게 여러 모델을 연결하고, **Language Server Protocol(LSP)** 기반 코드 이해와 멀티 세션 흐름을 강조합니다. CLI 중심이지만 desktop과 IDE 표면도 함께 가져가려는 점이 흥미롭습니다. 즉, “터미널이 중심이지만 터미널만이 전부는 아니다”라는 현실적인 태도입니다.

[OpenHands](https://openhands.dev/)는 조금 더 플랫폼에 가깝습니다. CLI, GUI, SDK, cloud/self-hosted 흐름을 함께 제공하면서 에이전트를 제품 환경에 넣는 쪽으로 유명해졌습니다. 터미널 하네스라기보다 에이전트 운영체제에 가까운 느낌입니다. 그래서 OpenHands를 “그냥 CLI 도구”라고 부르면 조금 좁게 보는 것입니다.

[Aider](https://aider.chat/)는 반대로 아주 선명합니다. Git-native한 터미널 페어 프로그래머입니다. 완전 자율 에이전트라기보다는 사람과 같이 앉아 diff를 만들고, 커밋 단위로 사고하고, 레포지토리 맵을 활용하는 쪽에 강합니다. 유명해진 이유도 여기에 있습니다. 거창한 오케스트레이션보다, 개발자가 매일 쓰기 좋은 마찰 없는 루프를 만들었습니다.

[SWE-agent](https://swe-agent.com/latest/)는 연구와 벤치마크의 언어로 유명해졌습니다. GitHub 이슈를 받아 레포지토리를 고치고 테스트로 검증하는 구조는 SWE-bench 시대의 상징적인 하네스였습니다. 다만 현재는 주요 개발 흐름이 mini-swe-agent 쪽으로 이동하고 있다는 점을 함께 봐야 합니다. 여기서 중요한 교훈은 이름이 아니라 패턴입니다. 이슈를 읽고, 환경을 구성하고, 실패를 관찰하고, 패치를 검증하는 루프가 코딩 에이전트의 기본 문법이 되었습니다.

Nous Research의 [Hermes Agent](https://github.com/NousResearch/hermes-agent)는 또 다른 방향을 보여 줍니다. 여기서 Hermes는 단순한 모델 이름이 아니라, 메모리와 skills, messaging gateway를 가진 자율 에이전트 시스템입니다. 터미널의 `hermes` 명령으로 시작할 수 있지만, Telegram, Discord, Slack 같은 메시징 표면으로도 확장됩니다. 이것이 흥미로운 이유는 하네스가 더 이상 “코딩 툴”에만 머물지 않고, 에이전트가 여러 채널에서 지속적으로 일하는 런타임으로 넓어지고 있기 때문입니다.

그리고 이 위에 더 opinionated한 래퍼들이 올라옵니다. [Oh My OpenAgent](https://ohmyopenagent.com/docs)는 OpenCode 위에서 parallel specialist agents, LSP, AST-Grep, hash-anchored edits, ultrawork 같은 워크플로우를 강조합니다. [Oh My Claude](https://github.com/TechDufus/oh-my-claude)는 Claude Code를 확장하는 플러그인으로, subagents, guardrails, review gates, ultrawork 스타일 오케스트레이션을 제공합니다. 둘 다 “새로운 모델”이 아니라 “일을 시키는 방식”으로 유명해진 사례입니다.

이 도구들을 한 줄로 정리하면 이렇게 볼 수 있습니다.

| 하네스 | 유명해진 이유 | 성격 |
| --- | --- | --- |
| Claude Code | 터미널-native 코딩 루프와 Anthropic 공식 경험 | 기준점에 가까운 상용 CLI 에이전트 |
| OpenCode | provider-agnostic, LSP, 멀티 세션 | 열린 터미널/desktop/IDE 에이전트 |
| OpenHands | CLI/GUI/SDK/cloud까지 포함한 플랫폼 | 에이전트 운영체제에 가까움 |
| Aider | Git-native 페어 프로그래밍 | 사람과 함께 가는 터미널 assistant |
| SWE-agent | SWE-bench와 issue-to-fix 루프 | 연구/벤치마크형 하네스 |
| Hermes Agent | memory, skills, messaging gateway | 지속 실행되는 자율 에이전트 시스템 |
| Oh My OpenAgent / Oh My Claude | parallel agents, review gates, ultrawork | 기존 CLI 에이전트를 강화하는 opinionated 플러그인 |

핵심은 이들이 모두 다른 방식으로 같은 질문에 답한다는 점입니다. “모델에게 무엇을 말할까?”가 아니라 “모델이 어떤 환경에서, 어떤 도구로, 어떤 루프를 돌며, 어떤 증거를 남기게 할까?”라는 질문입니다.

다시 말해, 모델이 혼자 똑똑한 것보다 모델이 일을 할 수 있게 만들어 주는 무대가 중요합니다. 그리고 현재 그 무대의 가장 실용적인 형태 중 하나가 CLI입니다.

## 검은 화면에서 일어나는 라이프스타일 변화

터미널 기반 에이전트 워크플로우를 며칠만 써 보면 묘한 생활감의 변화가 생깁니다.

예전에는 개발자가 계속 키보드를 두드렸습니다. 파일을 열고, 이름을 바꾸고, 테스트를 돌리고, 로그를 읽고, 다시 코드를 고쳤습니다. 이제는 더 자주 목표를 선언합니다.

“이 실패 테스트를 끝까지 고쳐.”

“이 API 변경을 호출부까지 반영해.”

“이 기능을 작은 커밋 단위로 나눠서 구현 계획을 세워.”

그리고 기다리는 동안 개발자는 커피를 마십니다. 물론 커피를 마신다고 책임이 사라지는 것은 아닙니다. 오히려 리뷰의 중요성은 더 커집니다. 에이전트가 빠르게 만든 코드는 빠르게 검증해야 합니다. 테스트, diff, 로그, 보안 관점, 설계 의도는 여전히 사람의 책임입니다.

다만 노동의 질감이 바뀝니다. 직접 벽돌을 하나씩 쌓는 시간은 줄어들고, 어떤 벽을 왜 세워야 하는지 판단하는 시간이 늘어납니다. 개발자는 타이피스트에서 현장 감독에 가까워집니다. 좋은 현장 감독은 삽을 들 줄도 알지만, 매 순간 직접 삽질하지는 않습니다.

이 변화가 터미널에서 더 잘 느껴지는 이유는 모든 것이 기록으로 남기 때문입니다. 명령, 결과, 실패, 재시도, 패치. 에이전트의 작업이 검은 화면 위에 층층이 쌓입니다. 채팅창에서는 “대화”처럼 보이던 것이 CLI에서는 “작업 로그”가 됩니다.

그리고 개발자에게 작업 로그는 꽤 강력한 안심 장치입니다.

## 좋은 하네스가 가져야 할 것들

그렇다면 좋은 CLI 에이전트 하네스는 무엇을 가져야 할까요? 기능 목록처럼 길게 쓰고 싶지는 않지만, 최소한 몇 가지는 중요합니다.

첫째, 도구 접근입니다. 에이전트는 파일을 읽고, 검색하고, 테스트를 실행하고, Git 상태를 확인할 수 있어야 합니다. 눈과 손이 없는 모델은 결국 말만 잘하는 컨설턴트가 됩니다. 컨설턴트도 좋지만, 가끔은 누가 빌드 좀 고쳐 줬으면 합니다.

둘째, 안전한 실행 환경입니다. 에이전트는 실수할 수 있습니다. 그래서 샌드박스, 권한 제어, 변경 범위 제한, 명시적 승인 지점이 필요합니다. 빠른 도구일수록 브레이크가 좋아야 합니다.

셋째, 반복 루프입니다. 한 번 생성하고 끝나는 방식은 실제 개발과 맞지 않습니다. 테스트가 실패하면 원인을 분석하고, 수정하고, 다시 실행해야 합니다. 이 루프를 얼마나 잘 설계하느냐가 하네스의 실력입니다.

넷째, 컨텍스트 관리입니다. 규칙, 과거 결정, 프로젝트 구조, 코딩 스타일, 금지 사항, 목표 상태가 에이전트에게 안정적으로 전달되어야 합니다. 컨텍스트가 흐릿하면 결과도 흐릿해집니다.

마지막으로, 리뷰 가능한 증거입니다. 에이전트가 “다 했습니다”라고 말하는 것만으로는 부족합니다. 어떤 테스트를 돌렸는지, 어떤 파일을 바꿨는지, 어떤 실패를 확인했는지 남아 있어야 합니다. 에이전트 시대의 신뢰는 말투가 아니라 증거에서 옵니다.
