# Project Glasswing: Claude Mythos Preview — 사이버 역량, 평가 지표, 그리고 개발자의 다음 단계

안녕하세요? HyperAccel ML팀 소속 박현준입니다.

2026년 4월 초, Anthropic은 [Project Glasswing](https://www.anthropic.com/glasswing) 이라는 산업 연대와 함께 **Claude Mythos Preview** 라는 미공개 프론티어 모델을 공개했습니다. 최근 **Large Language Model(LLM)** 은 코딩 보조를 넘어 에이전트형 워크플로까지 빠르게 확장했는데요. 동시에 [Frontier Red Team 블로그](https://red.anthropic.com/2026/mythos-preview/) 에서 한 달간의 내부 평가와 취약점 사례를 아주 길게 풀었습니다. 보통 최신 모델이 나오면 “벤치마크 점수가 올랐다” 정도로 끝나는데, 이번에는 이유가 분명합니다. Mythos Preview 수준의 코딩·추론 수준이 **취약점 탐지와 공격 코드 작성** 까지 한 번에 끌고 올라왔기 때문입니다.

그래서 이 글에서는 세 가지를 순서대로 다루려고 합니다.

1. Mythos Preview가 **무엇이 달라졌는지** (숫자와 사례 중심)
2. 그걸 보여 주기 위해 **평가 지표가 어떻게 진화** 했는지
3. **ML·시스템 엔지니어** 가 앞으로 무엇을 바꿔야 하는지

---

## 분기점: 왜 “최신 모델”을 제한적으로만 내놓았을까

Glasswing은 AWS, Apple, Google, Microsoft, Linux Foundation 등이 이름을 올린 **방어 우선(defense-first)** 협력 프로그램입니다. Mythos Preview 자체는 **일반 공개(GA) 계획이 없다** 고 Anthropic이 못을 박아 “방어 쪽이 먼저 손에 넣게” 설계했습니다.

Red Team 글에 따르면 Mythos Preview는 사용자가 지시할 경우 **주요 운영체제와 주요 웹 브라우저** 에서 **제로데이(zero-day)** , 즉 아직 공개되지 않은 취약점을 찾아내고 공격 코드까지 만들 수 있는 수준이라고 합니다. Anthropic이 제한 공개를 택한 이유도 여기에 있습니다. 이런 능력이 빠르게 퍼질 수 있다고 보고, **책임 있는 공개(coordinated disclosure)** 절차를 먼저 깔아 둔 것입니다.

한 줄로 요약하면, “AI가 코드를 잘 짠다”는 이야기를 넘어서 **“AI가 보안 연구자의 일부 역할을 자동화한다”** 는 단계에 들어섰다는 선언에 가깝습니다.

## Mythos Preview는 무엇이 다른가

### 명시적 ‘해킹 특화’ 훈련이 아니라도 능력이 붙는다

Red Team 글의 핵심 주장 중 하나는, Mythos Preview를 **사이버 공격용으로 따로 훈련하지 않았다** 는 점입니다. 대신 코드 이해, 추론, 자율 실험 같은 **일반 능력이 오르면** 패치도 잘하고, 그만큼 익스플로잇도 잘하게 된다는 이야기입니다. 즉 “방어에 좋은 것”과 “공격에 쓰일 수 있는 것”이 같은 엔진에서 나온다는 뜻이죠.

### 숫자로 보는 도약 (Opus 4.6 대비)

Glasswing 페이지와 Red Team 글에 나온 비교만 골라 적어 보겠습니다.

- **CyberGym** (취약점 재현): Mythos Preview **83.1%**, Opus 4.6 **66.6%**
- **SWE-bench Verified**: Mythos **93.9%**, Opus 4.6 **80.8%**
- **Terminal-Bench 2.0**: Mythos **82.0%**, Opus 4.6 **65.4%**

숫자만 보면 감이 잘 안 옵니다. 실제 사례로 보면 차이가 더 선명합니다.

- **OSS-Fuzz 내부 평가** (심각도 1~5): 약 7000개 진입점에 한 번씩 시도했을 때, Opus 4.6/ Sonnet 4.6은 최고 단계(제어 흐름 탈취)에 거의 도달하지 못했지만, Mythos는 **10건** 을 만들었습니다.
- **Firefox 147 실험** (연구용 환경, 실제 브라우저 샌드박스 제외): Opus 4.6은 수백 번 시도해 **2번** 성공했지만, Mythos는 **181번** 성공했고 **29번** 은 레지스터 제어까지 갔습니다.

즉 “조금 더 잘한다” 정도가 아니라, 사람이 오래 매달려야 찾을 문제를 모델이 훨씬 빠르게 찾아내는 단계로 넘어갔다고 보는 편이 더 자연스럽습니다.

## 벤치마크는 어디로 가고 있는가

Anthropic이 [Claude Opus 4.7](https://www.anthropic.com/news/claude-opus-4-7) 을 내놓으면서 함께 실은 평가 표는, “모델이 얼마나 똑똑한가”를 한 축으로만 보지 않습니다. 코딩 에이전트, 터미널 작업, 긴 맥락, 도구 사용, 금융·문서 같은 **도메인 벤치** 까지 나열되어 있습니다.

벤치마크의 흐름을 세대로 나누어 보면 다음과 같이 정리할 수 있습니다.

1. **패턴 맞추기에 가까운 시험** — 짧은 지식 문제, 작은 함수 하나 고치기 정도가 가능한지 평가합니다.
2. **실제 코드베이스 단위의 코딩** — 파일 한 조각이 아니라 프로젝트 전체 문맥을 보고 문제를 푸는지 체크합니다.
3. **장기 에이전틱 실행** — 한 번의 답을 맞추는 것이 아니라 **시도·실패·재시도** 를 포함하여 더 고수준의 문제를 해결할 수 있는지 확인합니다.
4. **실전 결과 중심 평가** — Mythos 문서가 강조한 방향입니다. 공개 벤치에 없는 **제로데이** (아직 공개되지 않은 취약점)를 찾으면 “외워서 푼 것”과 구분할 수 있고, 이 부분에서 기존 모델과 차별화된 가치가 생깁니다.

Red Team 글은 Mythos가 기존 벤치마크들을 100점에 수렴시키기 시작했다고 말합니다. 그래서 측정의 중심이 “점수표”에서 **“실제 코드베이스에서 무엇을 해냈는가”** 로 옮겨가고 있습니다.

## 방어 쪽에서 실제로 잡아낸 것들

아래는 Red Team 글과 Glasswing 소개에 **이미 공개·패치된 사례** 로 나온 것들입니다.

### OpenBSD, TCP SACK 쪽 27년 묵은 취약점

TCP의 **Selective Acknowledgment(SACK)** 구현에서, 홀(hole) 리스트를 갱신할 때 **시작 범위 검증이 빠진 버그** 와, 그 결과 **부호 있는 정수 오버플로** 로 “원래 동시에 성립할 수 없는” 분기가 열리는 두 번째 버그가 겹쳤습니다. 원격에서 머신을 크래시시키는 **Denial of Service(DoS)** 로 이어집니다.

Mythos는 격리 컨테이너 안에서 소스를 읽고, 가설을 세우고, 실제로 돌려 보며 **Address Sanitizer(ASan)** 같은 오라클로 확인하는 **에이전틱 스캐폴드** 로 버그를 냈습니다. OpenBSD처럼 오래 갈고 닦인 코드에서도, “사람이 한 줄씩 의심하기엔 너무 넓은 표면”을 **병렬 탐색** 으로 덮는 그림이죠.

### FFmpeg, H.264 디코더 16년 묵은 취약점

슬라이스 번호는 32비트로 올라가는데, 매크로블록 위치 테이블은 16비트로만 기록됩니다. `memset` 으로 테이블을 `-1` 로 채워 **센티널 65535** 를 쓰는데, 공격적으로 슬라이스 수를 끝까지 밀면 **센티널과 진짜 슬라이스 번호가 충돌** 합니다. 그 결과 이웃 매크로블록을 잘못 참조해 힙에 몇 바이트 튀어나가는 식의 문제로 이어집니다.

흥미로운 점은, 자동화된 퍼저가 해당 줄을 **수백만 번** 때렸다는 사실입니다. 즉 “더 많이 때리기”만으로는 안 잡히던 클래스를, 모델이 **의미와 불변 조건** 쪽에서 읽어낸 사례로 읽을 수 있습니다.

### FreeBSD NFS, **Remote Code Execution(RCE)** (CVE-2026-4747)

**Remote Procedure Call(RPC)** 의 보안 확장인 RPCSEC_GSS 경로에서 공격자 패킷을 짧은 스택 버퍼로 복사하는데, 길이 검사가 느슨해 **큰 스택 오버플로** 가 납니다. 여기에 스택 카나리가 없는 컴파일 경로, 커널 **Address Space Layout Randomization(ASLR)** 이 약한 점 등이 겹쳐 **ROP** 로 `/root/.ssh/authorized_keys` 에 공격자 키를 붙이는 익스플로잇까지 이어졌다고 합니다. 긴 체인은 **여러 RPC 요청으로 쪼개** 넣었고요.

Glasswing과 Red Team 글 모두에서 “사람 개입 없이, 초기 프롬프트 이후 **완전 자율**” 이라는 점을 강조합니다. 비교 대상으로, 외부 연구 회사가 Opus 4.6으로 같은 취약점을 익스플로잇했을 때는 **사람의 안내가 필요했다** 고도 적혀 있습니다.

### Linux 커널, 여러 취약점 체이닝

단일 버그만으로는 **Kernel ASLR(KASLR)** 같은 방어를 못 뚫을 때가 많습니다. Mythos는 읽기 버그로 KASLR을 깨고, 쓰기 버그와 **Use-After-Free(UAF)** , 힙 스프레이를 조합해 **로컬 권한 상승** 체인을 여러 개 만들었다고 합니다. 일부는 아직 공개 전이라 세부는 SHA-3 커밋먼트로만 남겨 두었죠.

이 네 가지를 한 줄로 묶으면, Mythos Preview는 “코드를 잘 읽는 모델”을 넘어서 **검증 가능한 실패** 를 반복하며, 필요하면 **익스플로잇으로 심각도를 증명** 하는 쪽까지 간다는 뜻입니다. 방어자에게는 **같은 도구로 패치와 재현 스크립트를 앞당길 기회** 이고, 공격자에게는 CVE 공지 이후·전면 패치 이전 사이를 노리는 **N-day** 공격 창이 압축된다는 경고이기도 합니다. Anthropic이 말하듯, 패치가 공개되면 그 자체가 **로드맵** 이 되기 때문입니다.

## 개발자에게 남는 숙제

### 개발자: 코드 리뷰, 플랜 리뷰 다음은 무엇인가

[에이전트 도입기](/posts/how-we-use-ai/) 에서도 썼듯이, 에이전트로 코드 생산성이 오르면 먼저 병목이 되는 곳은 리뷰입니다. 사람이 코드를 줄 단위로 읽는 방식은 금방 한계가 와서, 일부 회사는 `plan.md` 를 먼저 보는 플랜 리뷰로 옮겨가고 있습니다.

다만 이 분야는 변화 속도가 너무 빠르기에 “어떤 리뷰 방식이 정답이다”라고 지금 단정하기는 어렵다고 봅니다.

핵심은 하나의 고정된 프로세스를 선언하는 것이 아니라, 에이전트 성능의 다음 점프에 맞춰 리뷰 방식도 빠르게 갱신할 준비를 갖추는 일입니다. 저희팀은 이미 플랜 리뷰 단계까지는 넘어왔고, 앞으로 출시될 에이전트 역량에 맞춰 프롬프트, 하네스, 검증 장치를 유기적으로 바꿔 갈 계획입니다. 특히 모델과 세션 단위로 권한을 제한하고, 무한 루프나 공회전에 빠지지 않도록 가드레일을 두고, 중단 조건을 명확히 정의하며, 에이전트가 실제로 수행한 작업을 사람이 바로 인식할 수 있게 실행 로그와 중간 산출물을 촘촘히 남기는 쪽으로 하네스를 설계하려고 합니다.
