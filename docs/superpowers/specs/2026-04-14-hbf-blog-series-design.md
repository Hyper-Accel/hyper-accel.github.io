# HBF 기술 블로그 시리즈 설계서

## 개요

HBF(High Bandwidth Flash) 기술을 다루는 3부작 시리즈 블로그 글의 설계서.
이 문서는 **1편** 의 상세 구조와, 2-3편의 방향성을 정의한다.

### 섹션별 구조

#### 섹션 1: 훅 & 도입
- HBF 관련주 급등 현상 언급
- "기술의 실체를 알아보자"
- "GPU 메모리는 왜 이렇게 비싸고, AI 서버는 왜 메모리 때문에 병목인가?"
- 이 글에서 다룰 내용 로드맵 + 시리즈 소개

#### 섹션 2: 메모리는 왜 여러 종류인가 — 속도/용량/비용의 트릴레마
- 핵심 명제: 빠른 메모리는 비싸고 작고, 싼 메모리는 느리고 크다
- 이것이 물리적 셀 구조에서 비롯됨을 명시
- 이후 섹션들의 프레임 역할

#### 섹션 3: SRAM — 가장 빠르고, 가장 비싼
- 셀 구조: 6개 트랜지스터(6T)로 1비트 저장
- 왜 빠른가: 트랜지스터 래치로 상태 유지, 리프레시 불필요 (Static), 읽기가 곧 전압 확인
- 왜 비싼가: 트랜지스터 6개 = 셀 면적 큼 → 같은 웨이퍼에서 bit 수 적음 → bit당 단가 높음
- 사용처: CPU/GPU 내부 캐시(L1/L2/L3), 가속기 온칩 메모리
- 수치: 접근 지연 ~1ns 이하, GPU 내 수 MB~수십 MB 수준

#### 섹션 4: DRAM — 싸고 크지만, 끊임없이 새로고침
- 셀 구조: 트랜지스터 1개 + 커패시터 1개 (1T1C)
- 왜 싼가: 셀 면적이 SRAM의 1/4~1/6 → 같은 면적에 훨씬 많은 비트
- 왜 느린가(상대적): 커패시터 충/방전 시간, 주기적 리프레시 필요 (Dynamic - 전하 누설)
- DDR 발전: 버스 폭과 클럭으로 대역폭 확보, 핀 수 한계 (DDR5 = 64-bit 버스)
- 수치: 접근 지연 ~10-100ns, 단일 모듈 수 GB~수십 GB

#### 섹션 5: HBM — DRAM을 쌓아올리다
- 핵심: 같은 DRAM 셀(1T1C)인데 패키징이 다르다
- TSV(Through-Silicon Via): 실리콘 관통 전극으로 다이를 수직 연결
- 와이드 I/O: 1024-bit 버스 (DDR5의 16배) → 대역폭 폭증
- 인터포저: GPU/가속기와 같은 기판 위에 나란히 배치 → 배선 거리 최소화
- 왜 비싼가: TSV 공정, 인터포저, 수율 관리, 열 문제 → 용량 확장에 물리적 한계
- 수치: HBM3E ~1.2 TB/s, 스택당 24-36GB, HBM4 로드맵
- 한계: 용량 대비 가격이 매우 높음, AI 모델 크기 증가 속도를 못 따라감

#### 섹션 6: NAND Flash(SSD) — 싸고 크지만, 너무 느린
- 셀 구조: 플로팅 게이트 트랜지스터, 전하 가둬서 비트 저장
- 왜 싸고 큰가: 비휘발성(전원 꺼도 유지), TLC/QLC로 셀당 3-4비트 → 극한의 밀도
- 왜 느린가: 읽기 ~100us, 쓰기 더 느림, erase 필요 → 랜덤 접근 부적합
- 현재 AI 서버에서의 역할: 모델 저장소, 체크포인트 → 연산에 직접 참여 못함

#### 섹션 7: 빈 자리 — HBM과 SSD 사이의 간극
- 비교 표: SRAM / DRAM / HBM / ??? / NAND (대역폭, 지연시간, 용량, bit당 비용)
- "HBM의 대역폭은 좋지만 용량이 부족하고, SSD는 용량은 넉넉하지만 너무 느리다"
- 이 간극을 채울 수 있다면? → HBF 소개로 자연스럽게 연결

#### 섹션 8: HBF — NAND에 HBM의 옷을 입히다
- 핵심 컨셉: NAND 셀은 그대로, HBM의 패키징 기술(TSV + 인터포저)을 적용
- SanDisk CBA(CMOS Bonding Array) 아키텍처: NAND를 저지연/고대역폭 접근에 최적화
- 병렬 구조: 수천 개의 독립적인 스토리지 서브어레이가 동시 작동
- Gen 1 스펙: 읽기 1.6 TB/s, 512GB (256Gb/die x 16 stack)
- Gen 2/3 로드맵: >2 TB/s / 1TB, >3.2 TB/s / 1.5TB
- HBM 대비: 8-16배 용량, 유사 대역폭, 유사 비용
- HBF PHY가 HBM과 호환 → 기존 가속기 HBM 컨트롤러에 직접 연결 가능
- 한계: 지연시간 ~10us (HBM ~100ns 대비 100배), 쓰기 속도 느림, write endurance 제한

#### 섹션 9: 마무리 & 2편 티저
- 1편 핵심 요약: 메모리 트릴레마, HBF가 채우는 빈 자리
- 2편 예고: 메모리 벽(Memory Wall), 삼성/하이닉스/SanDisk 경쟁 구도
- HyperAccel 채용 CTA

## 팩트체크 결과 요약 (글 작성 시 참조)

### 검증된 팩트
- HBF = High Bandwidth Flash, SanDisk + SK하이닉스 공동 개발/표준화
- 2025.08: MoU 체결, 2026.02: JEDEC 표준화 컨소시엄 Kick-Off
- Gen 1: 1.6 TB/s 읽기, 512GB (16 die stack)
- Gen 2: >2 TB/s, 1TB / Gen 3: >3.2 TB/s, 1.5TB
- HBM 대비 8-16x 용량, 유사 대역폭/비용
- 지연시간 ~10us (HBM ~10-100ns)
- H2 2026 샘플, 2027 초 AI 추론 서버 탑재 목표
- SK하이닉스 H3 아키텍처: HBM+HBF 하이브리드, 성능/와트 2.69x 향상 (IEEE 논문)
- 삼성: 초기 컨셉 설계 단계, 마이크론: 공식 HBF 참여 미확인

### 주의할 충돌 사항
- 스택당 대역폭 수치: SanDisk 공식 1.6 TB/s vs 일부 출처 400-800 GB/s (측정 기준 차이 가능)
- 시장 진입 시점: 샘플 2027 vs 본격 양산 2030 (시점 정의 차이)
